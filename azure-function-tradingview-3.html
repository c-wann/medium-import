Using Azure Function to Trade on TradingView (3/3)

Your bot now executes every TradingView signal without hesitation — which is exactly the problem. A crossover in a choppy, low-volume market deserves a second opinion. In this final part we add that second opinion by wiring a Microsoft Foundry AI agent into the pipeline, giving the bot a reasoning layer that decides whether a signal is worth acting on before the order ever reaches Binance.

Why Add AI Reasoning to a Trading Bot?

Rule-based strategies are fast and deterministic, but they are blind to context. An EMA crossover fires the same way during a strong trend as it does during a noisy sideways market. A human trader would glance at volume, the broader market direction, and recent news before clicking buy — an AI agent can do the same thing programmatically in milliseconds.

Rather than replacing the strategy, the agent acts as a gatekeeper. The Azure Function that was called directly by TradingView in Part 2 becomes an internal tool that the Foundry agent decides when to invoke. The agent can consult real-time web data, apply reasoning rules, and either approve the trade or suppress it with a logged explanation — all without touching the Pine Script or the Binance integration you already built.

What is Microsoft Foundry?

Microsoft Foundry is a unified Azure platform-as-a-service for building production AI applications. It consolidates models, agents, tools, and observability under a single resource — replacing the older split between Azure OpenAI and Azure AI Studio. The platform gives you access to over 1,900 models (GPT-5, Claude, Mistral, Phi-4, and more) and a tool catalog with over 1,400 integrations, all governed through a single Azure resource with role-based access control and built-in tracing.

The key capability for our scenario is the Foundry Agent Service tool catalog, which includes a native Azure Functions built-in tool. This means a Foundry agent can call your existing Azure Function directly — no new API layer needed. The agent decides when to call it based on its instructions and the incoming signal data.

Creating a Foundry Agent with the Azure Functions Tool

Start by creating a Foundry resource in the Microsoft Foundry portal. Create a new project, then navigate to Build → Agents and click New agent. Choose gpt-4.1 or gpt-4.1-mini as the model — the mini variant is fast and cost-effective for this kind of structured decision task.

Give the agent a system prompt that describes its role as a trade validator. A starting point: "You are a cautious trading assistant. You receive trade signals containing a symbol, side (BUY/SELL), price, and total. Before placing an order, use the web search tool to check if there is any breaking news or unusual volatility for the symbol. If conditions look normal, call the place_order function. If risk is elevated, respond with a brief explanation of why the trade was suppressed."

Next, add two tools to the agent: Web Search (a built-in tool — toggle it on in the Tools panel) and your Azure Function via the Azure Functions built-in tool. For the Azure Functions tool, provide the function app resource ID and select the http_bnb_limit_order function. Foundry handles authentication through your project's managed identity, so no API keys are passed in the prompt.

Wiring Foundry Into the Trading Pipeline

With the agent configured, update your TradingView webhook URL. Instead of pointing directly at the Azure Function, point it at a new thin HTTP trigger in the same Function App — call it http_signal_intake. This intake function receives the TradingView JSON payload and forwards it to the Foundry agent using the Microsoft Foundry Python SDK. 

import os
import azure.functions as func
from azure.identity import DefaultAzureCredential
from azure.ai.projects import AIProjectClient

app = func.FunctionApp(http_auth_level=func.AuthLevel.FUNCTION)

PROJECT_ENDPOINT = os.environ["FOUNDRY_PROJECT_ENDPOINT"]
AGENT_NAME       = os.environ["FOUNDRY_AGENT_NAME"]

@app.route(route="http_signal_intake")
async def http_signal_intake(req: func.HttpRequest) -> func.HttpResponse:
    body = req.get_json()
    if body is None:
        return func.HttpResponse("Invalid JSON", status_code=400)

    project = AIProjectClient(
        endpoint=PROJECT_ENDPOINT,
        credential=DefaultAzureCredential(),
    )
    openai = project.get_openai_client()

    user_message = (
        f"New trade signal received: symbol={body['symbol']}, "
        f"side={body['side']}, price={body['price']}, total={body['total']}. "
        "Validate and place the order if conditions are suitable."
    )

    response = openai.responses.create(
        extra_body={"agent_reference": {"name": AGENT_NAME, "type": "agent_reference"}},
        input=user_message,
    )

    return func.HttpResponse(response.output_text, status_code=200)

Add two new App Settings to your Function App: FOUNDRY_PROJECT_ENDPOINT (formatted as https://<resource-name>.ai.azure.com/api/projects/<project-name>) and FOUNDRY_AGENT_NAME. Enable a system-assigned managed identity on the Function App and grant it the Azure AI User role on the Foundry project. This allows DefaultAzureCredential() to authenticate without any stored secrets.

The Complete Autonomous Stack

The final architecture looks like this: TradingView fires a webhook → http_signal_intake receives it → the Foundry agent runs a web search for market context → the agent reasons about the signal → if approved, the agent calls http_bnb_limit_order → the order lands on Binance. Every agent decision is logged in Foundry's built-in tracing dashboard, giving you a complete audit trail of which signals were executed and which were suppressed and why.

This architecture separates concerns cleanly: Pine Script handles chart analysis, Azure Functions handle exchange integration, and Microsoft Foundry handles judgement. Each layer can be improved independently — you can swap in a more aggressive model during bull markets, add a second agent for risk sizing, or hook in the Code Interpreter tool to run statistical checks on recent candle data before any order is placed.

The platform is free to explore, and you only pay for model tokens consumed — GPT-4.1-mini costs a fraction of a cent per signal evaluation at typical prompt lengths. For a bot trading modest position sizes, the AI layer adds negligible cost while dramatically reducing the risk of entering bad trades during high-volatility, news-driven moves that a pure technical signal would miss entirely.
