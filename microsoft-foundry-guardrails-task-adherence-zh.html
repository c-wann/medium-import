Microsoft Foundry 的 AI 護欄：Task Adherence 如何防止代理人越權操作

當 AI 代理人被賦予執行真實操作的能力——發送電子郵件、修改資料庫、提交請假申請——一旦它誤解用戶意圖，後果可能比傳統軟體 bug 嚴重得多。Microsoft Foundry 的 Task Adherence（任務遵從性，預覽）功能就是為此而生：它在代理人呼叫工具之前，先確認操作是否與用戶的實際意圖一致。

什麼是 Task Adherence？

Task Adherence 是 Foundry Guardrails & Controls（護欄與管控）體系的一部分，屬於 Azure AI Content Safety 的代理人工作流程保護層。它分析：用戶的原始意圖（Information request？行動指令？確認？）、代理人計劃執行的工具呼叫（Write？Delete？Update？），以及工具的輸入/輸出是否與對話上下文一致，最終輸出一個 taskRiskDetected（布林值）與風險說明。

Task Adherence 的設計原則是：用戶說「查一下」，代理人不應該做「改一下」。這聽起來簡單，但在實際的對話流中，語言的模糊性讓 LLM 很容易做出超出用戶預期的操作。這個護欄提供一道獨立的判斷層，不依賴代理人的系統提示詞，而是由專門的 Content Safety 模型做二次確認。

三個典型場景

客戶服務情境：用戶問「我這個月用了多少數據流量？」代理人計劃呼叫 change_data_plan()。Task Adherence 判斷：用戶在請求資訊，但代理人計劃修改方案，風險偵測為 true，阻斷執行並回傳說明理由，系統可選擇提醒用戶確認或直接改為呼叫 get_data_usage()。

人資管理情境：員工問「我還有多少年假？」代理人計劃呼叫 apply_leave()。Task Adherence 識別任務不符——用戶問的是資訊查詢，代理人卻計劃發起請假流程。同樣阻斷執行，讓代理人改為呼叫 get_leave_balance()。

生產力工具情境：用戶說「幫我寫一封關於延誤交付的郵件給客戶。」代理人撰寫完郵件後計劃呼叫 send_email()。Task Adherence 判斷：用戶並未明確指示「發送」，只說了「寫」，因此標記為潛在風險，阻斷發送並改為讓用戶確認是否發出。這三個案例的輸出格式完全相同：{"taskRiskDetected": true, "details": "..."}，方便在任何後端處理。

整合 Task Adherence 的方式

在 Foundry Portal，進入您的專案，左側選 Guardrails + controls，點選 Try it out 頁面，找到 Agentic Workflow > Task Adherence。您可以直接用範例對話測試，也可以輸入自訂的用戶訊息和計劃工具呼叫，觀察 Task Adherence 的判斷結果。這個試驗頁面是在不寫任何程式碼的情況下了解護欄行為的最快方式。

在程式碼中，透過 Azure AI Content Safety REST API 或 SDK 可以在代理人的工具呼叫前插入 Task Adherence 檢查。典型的整合模式是：代理人規劃好要執行的工具 → 呼叫 Task Adherence API → 根據 taskRiskDetected 決定繼續、修改計劃或請求人工確認。詳細的 REST API 規格請見Azure AI Content Safety 文件。

技術限制與最佳實踐

Task Adherence 目前有幾個已知限制：最大輸入長度 100,000 字元；主要針對英文最佳化，其他語言效果可能不穩定；資料可能被路由到美國或歐盟地區以外的其他節點處理，需確認符合資料主權要求。功能目前是預覽（Preview）狀態，不建議用於關鍵生產工作負載，但適合在開發與測試環境中建立 AI 安全意識。

最佳實踐建議：不要只靠 Task Adherence 單一護欄；建議搭配 Prompt Shields（防止提示詞注入攻擊）、Content Safety（過濾有害內容），以及良好的代理人系統提示詞設計，形成多層防禦（Defense in Depth）。Task Adherence 是最後一道防線，而不是唯一一道。更多 Foundry Guardrails 的全貌請見Guardrails Overview 文件。
