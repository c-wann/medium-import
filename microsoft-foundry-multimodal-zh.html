Microsoft Foundry 多模態 AI：影像、語音與文字的整合開發實戰

文字不再是 AI 應用的唯一輸入。在 Microsoft Foundry 的模型目錄中，您可以取用 GPT-4.1（視覺+文字）、GPT-4o Audio（音訊+文字）、Phi-4-multimodal（視覺+語音+文字）、Stability AI SDXL（文字生成影像）等多模態模型，在同一個平台、同一個 SDK、同一個計費帳戶下建立處理圖片、聲音與文字的 AI 應用。本文帶您了解 Foundry 的多模態能力全貌與開發實戰。

視覺理解：讓 AI 看懂圖片

GPT-4.1 Vision 和 GPT-4o 都支援在同一個 API 呼叫中傳入文字和圖片。在 Responses API 中，圖片可以用兩種方式傳入：URL（公開可存取的圖片連結）或 Base64 編碼的圖片資料。Foundry Portal 的 Playground 支援直接拖拽圖片測試，不需要寫任何程式碼就能感受視覺理解的能力。

實際應用場景非常廣泛：工廠品質控制——上傳產品照片，讓代理人判斷是否有瑕疵；文件處理——傳入掃描的合約或報告圖片，讓模型提取關鍵資訊；醫療影像初篩——結合 RAG 知識庫，提供圖片與對應診斷指引；電商商品分析——根據商品照片自動生成商品描述和分類標籤。這些場景過去需要單獨的視覺 ML 模型，現在透過 Foundry 的多模態 LLM 可以在一個統一的 API 介面實現。

語音理解與語音合成

GPT-4o Audio（gpt-4o-audio-preview）支援直接傳入音訊 clip 並在回覆中包含音訊輸出，同一個模型既能「聽」也能「說」。使用 Audio Completions API 時，輸入可以是 MP3、WAV、FLAC、OGG 等格式的音訊，輸出可以指定回應語言、語調和速度。另外，Whisper（語音轉文字）和 TTS（Text-to-Speech）作為獨立模型繼續可用，適合只需要單向語音處理的場景。

Phi-4-multimodal 是微軟自家的輕量級多模態模型，在視覺和語音理解上表現出色，模型尺寸遠小於 GPT-4.1，部署成本更低，尤其適合需要在邊緣裝置（Edge Device）或對延遲要求較高的場景中使用。在 Model Catalog 中可以找到 Phi-4-multimodal 的詳細說明和 Serverless API 計費方式。

影像生成：從文字到圖片

Foundry 模型目錄提供多個影像生成模型，包括 DALL-E 3（OpenAI）、Stability AI SDXL、Flux（Black Forest Labs）等。影像生成透過 Images API 呼叫，傳入文字提示詞（Prompt）和圖片尺寸設定（如 1024x1024），模型返回生成的圖片 URL 或 Base64 資料。Foundry Portal 的 Playground 同樣支援視覺化測試，可以直接比較不同模型在相同提示詞下的生成效果。

在代理人工作流程中，影像生成可以作為一個工具（Custom Function Tool）整合：用戶描述需求，代理人理解意圖後自動呼叫影像生成 API，並在對話中返回圖片連結。結合 Code Interpreter（讓代理人處理 Python 程式碼），還能做到「生成圖片後根據用戶反饋修改」的互動式影像創作流程。

多模態 RAG：讓知識庫理解圖片文件

企業文件中包含大量圖表、示意圖和截圖，這些內容在傳統 RAG 管線中往往被忽略（因為純文字 Embedding 無法索引圖片內容）。Foundry 的 多模態 RAG 解決方案是：在建立知識庫索引時，使用支援視覺的 Embedding 模型或 GPT-4o Vision 為文件頁面（包含圖片）生成描述，再將描述納入向量索引。這樣包含圖表的技術手冊、產品目錄或設計文件就能被正確檢索，代理人能夠回答「圖表中顯示的數據是什麼」這類問題。

多模態 AI 開發在 Foundry 中的起點是 Model Catalog，找到您目標的多模態模型，先在 Playground 用幾個真實的輸入測試，確認符合需求後再建立部署。建議在早期測試階段使用 Serverless API 部署（按用量付費），規模化後再評估是否切換到 Provisioned Throughput（PTU，確保穩定吞吐量）部署模式。各多模態模型的詳細計費資訊請見Azure OpenAI 服務定價頁面。
